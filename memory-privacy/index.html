Pebble Memory & Privacy
Built to remember — built to protect

How Pebble learns your family.
And how we keep it safe.

Pebble is not a chatbot that forgets between sessions. It builds a living memory of your family — one that gets richer every week. This page explains exactly how that memory works, what feeds it, and the security measures we have put in place to make sure only your family can access it.

Why it matters

A family’s memory shouldn’t live in twelve places.

Notes apps, group texts, camera rolls, Google Calendar, the daycare app, the school portal — and whatever a parent is holding in their head at any given moment. Nothing talks to anything. Pebble’s job is to become the one place it all lives, and the intelligence that connects it: the contextual brain of the family.

Not AI parenting advice. AI operational relief.
Parents don’t want another expert telling them what to do. They want a family chief of staff — something that quietly carries the coordination load.

How memory is organised

Four layers of memory.

Not all memory is equal. PebblePath holds it in four layers — from bedrock facts that rarely change to live context refreshed by the hour. Each layer is captured differently, and trusted differently.

Anchors
Rarely changes
Identity and birthdays, allergies and medical notes, clothing & shoe sizes, the key people and places, the pediatrician. The bedrock everything else hangs on. Set once at setup and confirmed, never re-asked.
Examples: Felix · born Oct 2023 · peanut allergy · size 5T · Dr. Morrison at Lakeview Pediatrics
Rhythms
Recurring
Daily and weekly routines, schedules, recurring events — daycare drop-off at 8:15, swim on Tuesdays, bath on alternating nights, bedtime at 7:30. The shape of a normal week. Synced from calendars and confirmed from repetition.
Examples: Monday gym · Tuesday swim class · Wednesday library storytime · Friday Grandma pickup
Patterns
Earned over time
Preferences, lessons and tendencies — “Felix skips nap after swimming,” “morning flights travel better,” “Sundays tend to get overbooked.” The memory that feels like magic. Derived from behaviour, almost never asked.
Source: Synthesised nightly by Pebble from behavioral signals · never manually entered
Live context
Now → 2 weeks
This week’s calendar, tomorrow’s weather, what’s running low, today’s handoffs between parents, upcoming events and deadlines. The fast-moving surface Pebble builds the daily brief from. Streamed continuously and expires on its own.
Examples: Weather forecast · calendar imports · parent handoffs · active trips · unsigned forms

Rich memory, no overload

When Pebble asks — and when it doesn’t.

Data collection without overload. Every prompt Pebble considers generating passes through four filters. If any filter fails, Pebble stays quiet and infers instead of asking.

1

Can I infer this instead?

If the answer is already in an anchor, rhythm, or can be derived from calendar or weather data, Pebble uses that instead of interrupting the parent. Asking is always the last resort.

2

Will I use this within 7 days?

Capturing a fact that has no near-term relevance just adds noise. Pebble only asks when there is a concrete, identifiable reason the information will be useful in the next week.

3

Can this be answered in 10 seconds?

If answering requires more than a brief tap or short reply, the question is too heavy. Pebble keeps prompts bite-sized: a single tap, a number, a yes or no. Long forms are never Pebble’s style.

4

Have I already asked something today?

No more than one proactive capture prompt per family per day — regardless of how many things Pebble could ask. This rate limit is hard-coded, not a guideline. Quiet days are the default.

The result: families using PebblePath typically receive roughly one capture prompt per day — often none. The memory compounds from passive sources, not from interrogation.
How we protect it

Security built in layers, not bolted on.

Family memory is sensitive. We have built security measures at every layer — from how data moves between your phone and our servers, to what AI models are ever allowed to see, to what happens when you close the app on a shared device.

Live — enforced

Cloudflare edge security

Every request to pebblepath.ai passes through Cloudflare before it reaches our servers. Six Transform Rules enforce a set of HTTP security headers on every response. These cannot be bypassed by pointing directly at the origin — the origin is not publicly reachable.

Active headers on every response
Content-Security-Policy HTTP Strict Transport Security X-Frame-Options: DENY X-Content-Type-Options: nosniff Referrer-Policy Permissions-Policy
CSP prevents script injection. HSTS forces HTTPS for all future visits. X-Frame-Options stops the site from being embedded in an iframe on a third-party page. Permissions-Policy disables access to camera, microphone, geolocation, and payment APIs from the marketing pages (the app uses iOS system APIs, not browser APIs).
Live — enforced

Hardware attestation — App Attest

On iOS, every request to our Cloud Functions is signed by Apple’s App Attest framework — a hardware-level cryptographic proof that the request genuinely came from the PebblePath app running on a real, unmodified Apple device.

No third-party script, automation tool, or API client can produce a valid App Attest token without a physical Apple device running the real app binary.
On the Portal (web), Firebase App Check with reCAPTCHA Enterprise performs the equivalent role — scoring browser sessions for bot signals before allowing Cloud Function calls.
Requests without a valid attestation token are rejected at the Cloud Function layer before any data is read or written.
Live — enforced

API key never stored on device

The Anthropic API key that powers all Pebble features is not stored in the iOS app binary or anywhere on-device. Every AI request routes through a Firebase Cloud Function running on Google’s servers.

Extracting the .ipa from the App Store and reverse-engineering it will not yield any API key.
The iOS app is a thin delegator — it sends family context to the Cloud Function, which handles the Anthropic call and returns the result.
The key is stored as a Cloud Functions secret, encrypted at rest in Google Secret Manager and injected at runtime only.
Live — enforced

PII tokenisation — names never reach Anthropic

Before any family context reaches an AI model, all child and parent names are replaced with anonymous tokens. The model reasons about {{CHILD_NAME}} — never “Felix” or “Thomas.”

Tokenisation happens inside the Cloud Function before the Anthropic API call is made. Detokenisation happens on the response, inside the same function, before the result is returned to the app.
Applied to all five Pebble generators: tips, insights, the daily brief, the pediatrician summary, and the Pebble chat path.
Anthropic’s servers therefore never process or store any real name associated with your family.
Live — enforced

Private mode — server-side enforcement

When you send a private message in Pebble, a visibility array on the Firestore document controls who can read it. This is a database rule — not a UI feature.

The Firestore security rule requires the requesting user’s uid to appear in the message’s visibility array before allowing a read.
Even a co-parent on a modified or patched client cannot bypass this — the enforcement happens at Google’s infrastructure, not in the app.
Private messages are also excluded from the nightly Pebble memory synthesis pass, so they never enter the shared context layer.
Optional — on iOS

Biometric lock — Face ID & Touch ID

You can require Face ID or Touch ID every time the app opens — so handing your phone to a child, leaving it on a table, or sharing a device with a partner doesn’t expose your family’s data.

Default off. Opt in via Settings → Security → Require Face ID / Touch ID.
Uses Apple’s LocalAuthentication framework — no biometric data is ever stored by PebblePath. The device handles the check entirely.
Locks immediately on background. No grace period unless you choose to add one in Settings.
Live — monitoring

Firestore security rules & audit log

Every Firestore collection is protected by explicit security rules. Only authenticated family members can read documents they belong to. An audit log records sensitive admin actions — quota changes, member removals, data deletions.

The flat-family model (Phase 5, 2026-05-19) removed all staged fallback rules. Every read and write is now strictly gated on membership, with no legacy bypass paths.
Child data access is gated on Child.parentIds — being in the family group alone does not grant access to a child’s milestones, insights, or Pebble chat.
Audit log writes are server-only (Cloud Function), preventing client-side tampering.

Our commitments to your family

  • Your family’s data is never sold. We run no advertising, ever.
  • You can delete your account and all associated data, instantly, from within the app.
  • Only the people you explicitly invite can see your family’s data — enforced at database level.
  • AI models receive anonymous tokens, not your children’s real names or identifiers.
  • What Pebble knows is visible and editable in Settings → What Pebble knows. Nothing is hidden.
  • PebblePath collects only what is useful in the next two weeks. Live context expires automatically.
  • No proactive capture prompt fires more than once per day per family. Quiet is the default.
  • We notify you when significant privacy or security changes happen — not buried in a ToS update.
